# 工作流自动化分析

> 分析4个工作流中哪些可以用脚本自动化，哪些需要AI

---

## 工作流1: 定时采集

### 当前状态
- ✅ **已自动化**（crontab定时执行）
- 脚本: `v2/scripts/collector_cron.py`
- 调度: crontab每小时/每2小时/每3小时

### 人工介入点
| 场景 | 操作 |
|------|------|
| Cookie过期 | 用户重新登录Twitter/微博 |
| 采集失败告警 | 用户检查并修复 |

### 结论
**完全自动化，无需AI参与**

## 工作流2: 选题报告

### 当前状态
- ❌ **需要AI生成**
- 需要理解内容、判断价值、分类筛选

### 可自动化的部分

#### 1. 数据预处理（可脚本化）
```python
# 可脚本化：从JSON提取基础信息
- 读取x.jsonl/weibo.jsonl/rss.jsonl
- 去重（相同URL）
- 按时间过滤（24h/48h）
- 提取基础字段：标题、摘要、来源、链接
- 输出：候选新闻列表（CSV/JSON）
```

#### 2. 信源打标（可脚本化）
```python
# 可脚本化：根据来源域名/账号打标签
- 官方账号列表 → 打标"一手官方"
- 权威媒体列表 → 打标"一手媒体"
- 其他 → 打标"二手编译"
```

#### 3. 时效性过滤（可脚本化）
```python
# 可脚本化：按时间过滤
- 发布时间 > 48h → 标记"过期"
- 发布时间 > 24h → 标记"补充"
- 发布时间 <= 24h → 标记"主体"
```

### 必须AI做的部分

| 任务 | 为什么需要AI |
|------|-------------|
| 内容理解 | 判断新闻属于哪个模块（AI/芯片/手机/游戏...） |
| 价值判断 | 判断是否符合用户偏好（强偏好/弱偏好） |
| 品牌识别 | 手机新闻属于哪个品牌（苹果/三星/华为...） |
| 入选理由 | 撰写为什么选这条新闻 |
| 头条推荐 | 判断哪3条是今日最重要的 |

### 结论
**半自动化**：脚本做预处理，AI做判断和生成

**建议流程**：
```
脚本预处理（去重/过滤/打标）→ AI生成选题报告 → 人工确认
```

## 工作流3: 撰写日报

### 当前状态
- ❌ **需要AI生成**
- 需要理解全文、撰写洞察、格式化输出

### 可自动化的部分

#### 1. 全文抓取（可脚本化）
```python
# 可脚本化：批量抓取原文
- 读取选题报告中的URL列表
- 使用Jina Reader批量抓取
- 输出：原文内容缓存
```

#### 2. 模板填充（可脚本化）
```python
# 可脚本化：基础信息填充
- 读取template.html
- 填充标题、日期等固定占位符
- 填充新闻基础信息（标题、来源、链接）
```

#### 3. 数据标签提取（半脚本化）
```python
# 半脚本化：正则提取数字
- 从正文中提取：金额、百分比、数量、时间
- 但需要AI判断哪些是关键数据
```

### 必须AI做的部分

| 任务 | 为什么需要AI |
|------|-------------|
| 正文撰写 | 压缩原文到50-100字，保留核心事实 |
| 洞察撰写 | 基于全文+观点，写≤300字符深度分析 |
| 观点搜索 | 判断需要搜索哪些观点支撑 |
| 信源标注 | 判断是一手官方/一手媒体/二手编译 |
| 格式检查 | 确保符合template.html结构 |

### 结论
**半自动化**：脚本做抓取和基础填充，AI做撰写和洞察

**建议流程**：
```
脚本抓取全文 → AI撰写正文+洞察 → 脚本填充模板 → AI检查格式
```

## 工作流4: 发布流程

### 当前状态
- ⚠️ **半自动化**
- 有脚本，但需要确认步骤

### 可完全自动化的部分

#### 1. GitHub Push（可脚本化）
```bash
# 可脚本化
#!/bin/bash
cd /Users/yangliu/Documents/Claude\ Code/codebuddy
git add tech-daily-brief/brief/$1.html
git commit -m "Add daily brief $1"
git push origin main
```

#### 2. 邮件发送（可脚本化）
```bash
# 已有脚本
python scripts/send_email.py brief/YYYY-MM-DD.html
```

#### 3. 公众号同步（可脚本化）
```bash
# 已有脚本
cd raphael-publish && node publish-daily.mjs ../tech-daily-brief/brief/YYYY-MM-DD.html
```

### 需要人工/AI确认的部分

| 步骤 | 为什么需要确认 |
|------|---------------|
| 测试邮件 | 需要人工确认效果OK后再群发 |
| 群发邮件 | 涉及23人，需要最终确认 |
| 公众号发布 | 需要人工在后台确认草稿 |

### 结论
**半自动化**：脚本执行操作，人工确认关键步骤

**建议流程**：
```
脚本执行GitHub Push → AI发送测试邮件 → 人工确认 → 脚本群发 → 脚本同步公众号
```

## 总结

| 工作流 | 自动化程度 | 脚本可做的 | 必须AI/人工的 |
|--------|-----------|-----------|--------------|
| 1.定时采集 | ✅ 100% | 全部 | 仅异常修复 |
| 2.选题报告 | ⚠️ 30% | 数据预处理、信源打标、时效过滤 | 内容理解、价值判断、生成报告 |
| 3.撰写日报 | ⚠️ 40% | 全文抓取、模板基础填充 | 正文撰写、洞察撰写、格式检查 |
| 4.发布流程 | ⚠️ 70% | GitHub Push、邮件发送、公众号同步 | 测试邮件确认、群发确认 |
